# PRD: 占城大师 v2 — 开地块与品阶压制

> 本 PRD 取代 `doc/design/mvp-prd.md`（v1）。完整机制调研见 `doc/research/gameplay-v2.md`，架构决策见 `docs/adr/0001-v2-architecture-pivot.md`，领域词汇以 `CONTEXT.md` 为准。

## Problem Statement

v1 的"迷雾翻牌 + 步骑弓三角克制"玩法存在三个设计瓶颈：（1）翻牌机制让地图信息随游戏进行单调递减，后期无事可做；（2）三角克制让兵种选择变成"剪刀石头布"而非真正的资源分配决策；（3）5×7 地图太小，推线节奏过短，玩家没有"布局感"。需要在 25 天内（截止 2026-06-22）用新的核心循环替换 v1，同时保留 v1 已经验证可玩的"单人 vs AI 自动对战"外壳。

## Solution

v2 用"开地块 + 品阶压制"替换 v1 的"翻牌 + 三角克制"：

- **地图扩大到 9×12 + 湖水行**（约 100 格），提供真正的"前线 + 后方"空间感；
- **格子改为状态机**（未激活 → 可开 → 已开），让玩家持续有"可开地块"这一核心决策目标；
- **开地块采用混合池 + 二次概率**：价格档位（25/50/100/250）决定成本，图标（? / 头盔 / 盾牌 / 金属）决定产出类别，档位 + 图标联合决定具体建筑；
- **兵种改为 4 品阶 × 6 种**，不做类型克制，纯品阶压制 + 行为差异化（防守反击 / 越格打建筑 / AOE）；
- **AI 从固定行为树改为评分-选择算法**，提供普通/困难两档可感知的难度。

## User Stories

### 地图与开局

1. 作为玩家，我想看到一张 9 列 × 12 行的六边形地图，中央有 1 行不可通行的湖水分割带，所以我能感受到"前线 + 后方"的空间层次。
2. 作为玩家，我想在游戏开始时看到我方的 HQ 和围绕 HQ 的 6 个已开建筑格（兵种/箭塔/金矿的随机组合），所以我能基于开局阵容规划早期策略。
3. 作为玩家，我想在开局至少 70% 概率获得至少 1 个金矿，所以我不会因"完全没经济"的极端开局而瞬间陷入困境。
4. 作为玩家，我想看到 AI 对手同样拥有 HQ + 6 个开局建筑，所以我能感受到对称的公平性。

### 格子状态与开地块

5. 作为玩家，我想看到与我方领地相邻的未激活格子被"激活"为可开状态（显示图标和价格标签），所以我能识别当前可投资的目标。
6. 作为玩家，我想看到可开格子的图标（? / 头盔 / 盾牌 / 金属）和价格档位（25/50/100/250），所以我能在"高风险低价"和"稳定高价"之间做空间决策。
7. 作为玩家，我想点击可开格子后按标价扣费并随机得到一个建筑（或空），所以我能通过资源分配塑造阵容。
8. 作为玩家，我想在 25 金档的 ? 兵营槽上有 50% 概率"空"（扣费但不出建筑），所以我能体验"便宜赌局"的风险刺激。
9. 作为玩家，我想已开格子不可逆地保持当前状态，所以我的每次开地块决策都有真实成本。
10. 作为玩家，我想开地块的图标和价格在格子激活时就固定（不随时间刷新），所以我能做基于当前信息的决策而非等待刷新。

### 经济与资源

11. 作为玩家，我想我的 HQ 被动产出金币（12 金/6 秒），所以我在 0 金矿开局下仍能缓慢发展。
12. 作为玩家，我想每个金矿每 6 秒产出 10 金，所以我有动力尽早投资金矿建立经济优势。
13. 作为玩家，我想看到实时金币数值显示在 UI 上，所以我能判断当前可承担的决策。
14. 作为玩家，我想在金币不足时点击可开格子得到"金币不足"反馈（而不是默默无响应），所以我能明确知道为什么不能操作。

### 建筑类型

15. 作为玩家，我想开出 ? 兵营槽后得到一个持续按 CD 吐出兵种的永久产兵建筑，所以我的"兵营投资"能持续回报。
16. 作为玩家，我想开出的兵种品阶按档位概率表决定（25 金档偏绿、250 金档偏紫金），所以我能"买品质"。
17. 作为玩家，我想开出头盔后必出一个兵种（不会空），具体哪种按档位概率表二次随机，所以我能选择"稳定但略贵"的投资路径。
18. 作为玩家，我想开出盾牌后得到一个箭塔（50 金档或 100 金档各 50%），所以我能补强防线。
19. 作为玩家，我想开出金属后直接得到金矿（无二次随机），所以我能稳定投资经济。
20. 作为玩家，我想升级已有金矿/箭塔（三档，每档成本翻倍），所以我能把早期建筑用到后期。
21. 作为玩家，我想兵营建筑不可升级，只能靠"开更高档位的新兵营槽"提升兵种品质，所以我的阵容有迭代空间而非单点强化。

### 兵种与产出

22. 作为玩家，我想兵营和已开头盔格子持续按兵种 CD（绿 3s / 蓝 4.5s / 紫 6s / 金 9s）吐出对应兵种，所以我能看到"投资回报"的节奏感。
23. 作为玩家，我想看到 6 种兵种（骷髅/野狼/战士/弓手/骑兵/龙骑兵）各有不同的 HP/ATK/移速/射程组合，所以我能感受阵容的多样性。
24. 作为玩家，我想看到蓝阶单位（战士/弓手）自动在近战和远程之间切换（远程范围内优先远程，敌人进入 1.2 距离内切近战），所以我能享受"智能单位"的观赏性。
25. 作为玩家，我想看到金阶龙骑兵在远程 CD 期间进入近战模式，所以最强的单位也有节奏弱点。

### 单位行为与寻敌

26. 作为玩家，我想绿/蓝阶单位攻击离我方阵营最近的敌方目标（单位或建筑），而不是深入敌后追击，所以我能感受"防守反击"的推线节奏。
27. 作为玩家，我想紫阶骑兵无视路径上的单位和格子，直接飞到敌方建筑格子攻击，所以我能享受"空军突袭"的爽感。
28. 作为玩家，我想金阶龙骑兵远程模式优先攻击敌方单位、近战模式打最近目标，所以我能感受"多功能单位"的策略价值。
29. 作为玩家，我想己方单位可自由穿过其他己方单位（不挡路），所以我不会因自己的单位拥堵而卡死推线。
30. 作为玩家，我想单位使用 BFS 寻路到最近目标，不会在复杂地形上卡死，所以我的单位行为可预测。

### 战斗与建筑 HP

31. 作为玩家，我想战斗采用"品阶压制"而非"类型克制"，高品阶单位综合能力明显强于低品阶，所以我能感受"抽到金兵的爽感"。
32. 作为玩家，我想所有建筑（HQ/兵营/箭塔/金矿）都有独立 HP，可被近战和远程攻击，所以推线的终点是"摧毁建筑"而非单纯"踩格子"。
33. 作为玩家，我想箭塔自动攻击范围内的敌方单位（50 金档射程 3.5、100 金档射程 5.0），所以我能感受"防线构建"的策略。
34. 作为玩家，我想 HQ 自带攻击能力（≈ 100 金箭塔，射程 6.0-7.0），所以 HQ 不只是被动挨打的肉盾。
35. 作为玩家，我想单位阵亡时播放消散/爆炸粒子特效，所以我能清晰感知战斗结果。

### 领地与推线

36. 作为玩家，我想我方单位踩过敌方格子时该格子立即变为我方颜色，所以我能感受"推线"的视觉反馈。
37. 作为玩家，我想敌方建筑被摧毁时该格子变为敌方颜色，直到我方单位越过废墟占领后方才连锁变回我方，所以推线不会因一个建筑倒塌而瞬间逆转。
38. 作为玩家，我想建筑被摧毁后留下废墟（不可重建、不可通行），所以我的早期投资决策有长期后果。
39. 作为玩家，我想左上角看到"占领 X/100"进度条，所以我能随时了解战局占比。

### 胜利条件

40. 作为玩家，我想在敌方 HQ 血量归零时看到所有敌方格子瞬间变为我方颜色（0.5 秒波浪扩散特效）+ 胜利弹窗，所以我能感受"直胜"的爽感。
41. 作为玩家，我想在己方 HQ 被摧毁时看到失败弹窗，所以我能明确游戏结束。
42. 作为玩家，我想看到 3:00 倒计时显示在 UI 上，所以我能感知对局节奏。
43. 作为玩家，我想在倒计时归零时系统比较双方占领格子数，多者胜，所以"推不动 HQ"也有终局。
44. 作为玩家，我想在游戏结束后看到"重新开始"按钮，所以我能立刻进入下一局。

### AI 对手

45. 作为玩家，我想 AI 遵守和我完全相同的经济规则（开地块付同价、按同概率表出建筑），所以 AI 的胜利是"靠聪明"而非"靠作弊"。
46. 作为玩家，我想在"普通"难度下 AI 每 1.5 秒决策一次，决策偶尔选次优动作（模拟人类考虑不周），所以我能体验到"有挑战但可赢"的对局。
47. 作为玩家，我想在"困难"难度下 AI 永远选最优动作、会反制我的布局（例如我造金矿它就推那条线），所以我能体验到"必须完美发挥"的压力。
48. 作为玩家，我想 AI 也会升级金矿和箭塔（困难 AI 根据战况动态选、普通 AI 只升金矿），所以 AI 不会随时间被我"熬死"。
49. 作为玩家，我想 AI 能看到我方的建筑和单位布局（全图可见），所以 AI 的应对是真实的而非盲目的。
50. 作为玩家，我想在 MVP 阶段可选普通/困难两档难度，所以我能按自己水平选对手。

### UI 与反馈

51. 作为玩家，我想在 UI 上看到：我方金币、AI 金币、3:00 倒计时、占领 X/100、HQ 血条，所以我能掌握战局全貌。
52. 作为玩家，我想在可开格子上看到清晰的图标和价格标签（即使格子很小），所以我能在不 zoom-in 的情况下做决策。
53. 作为玩家，我想在已开格子上看到对应建筑图标（? / 盾牌 / 金属 / 头盔 + 兵种图标），所以我能一眼分辨阵容构成。
54. 作为玩家，我想在 HQ 格子上看到血量条，所以我能感知"HQ 被围攻"的紧急程度。
55. 作为玩家，我想在格子变色的瞬间看到颜色过渡或闪烁，所以领地变化有视觉节奏。

## Implementation Decisions

### 模块划分

按 v2 Rewrite Strategy（见 `AGENTS.md` 和 `docs/adr/0001-v2-architecture-pivot.md`），所有模块定义在 `initGameState()` 之前，按以下顺序插入：

| 模块 | 职责 | 接口 |
|------|------|------|
| **HexGrid** | 9×12 + 湖水行的六边形坐标数学、像素转换、点击拾取 | `axialToPixel(q, r)`, `pixelToAxial(x, y)`, `getNeighbors(q, r)`, `isLake(q, r)`, `findHexAt(x, y)` |
| **RNG** | Seeded 随机数生成，单例贯穿整个游戏 | `next()`, `nextFloat()`, `weightedPick(options, weights)` |
| **TileSystem** | 格子状态机（未激活 → 可开 → 已开）、混合池分配、二次概率表、激活判定 | `activate(hexId)`、`open(hexId) → BuildingResult`、`onTerritoryChanged()` |
| **UnitTypes** | 6 兵种 × 4 品阶的纯数据表（HP/ATK/移速/射程/CD/攻击方式） | `getUnitSpec(tier, kind)`, `getAttackMode(tier, kind)` |
| **BuildingTypes** | 建筑纯数据表（HQ/兵营/箭塔 50/箭塔 100/金矿/废墟）的 HP/ATK/射程/产金/升级成本 | `getBuildingSpec(type, level)`, `getUpgradeCost(type, level)` |
| **ResourceSystem** | 双方金币管理、HQ 产金 + 金矿产金、花费校验 | `addGold(player, amount, source)`, `trySpend(player, amount) → bool`, `tickIncome()` |
| **BarracksSystem** | 兵营/头盔格子按兵种 CD 吐出单位 | `tick()` → spawn units from all barracks tiles |
| **MoveUnits** | 单位移动 + BFS 寻路 + 品阶差异化寻敌 + 飞行越格 + 己方穿越 | `tick()` → update unit positions, target selection per tier |
| **CombatSystem** | 单位交战 + 建筑受击 + 品阶压制 + 远程/近战切换 + AOE + 死亡特效触发 | `tick()` → resolve all combats, emit death events |
| **TerritorySystem** | 格子归属、推线连锁、建筑摧毁连锁 | `claim(hexId, player)`, `onBuildingDestroyed(hexId)`, `countTerritory(player)` |
| **RuinSystem** | 废墟生命周期（建筑摧毁 → 废墟格，不可重建不可通行） | `createRuin(hexId)`, `isRuin(hexId)` |
| **VictorySystem** | HQ 血量追踪 + 3:00 倒计时 + 双重胜负判定（HQ 摧毁直胜 / 倒计时比格数） | `tick(dt)`, `onHQDamaged(player, amount)`, `checkVictory() → GameResult` |
| **UpgradeSystem** | 金矿/箭塔三档升级（成本、属性提升） | `tryUpgrade(hexId, player) → bool` |
| **AIBehavior** | 评分-选择算法、难度参数化、合法动作枚举、启发式打分 | `tick(gameState, difficulty) → Action[]` |
| **OpeningLayout** | 开局 HQ 周围 6 格随机填入建筑（≥1 金矿概率 70%） | `generateOpening(rng) → HexState[]` |
| **DeathEffect** | 单位死亡粒子特效（消散/爆炸，纯视觉） | `spawn(position, color)`, `tick(dt)`, `render(ctx)` |
| **WaveEffect** | HQ 被摧毁瞬间的 0.5s 波浪扩散特效 | `spawn(position, color)`, `tick(dt)`, `render(ctx)` |
| **UI / Render** | drawHex、renderGrid、renderBuildings、renderUnits、renderUI（金币/倒计时/HQ 血条/占领进度/价格标签） | `render(state)` |
| **ClickHandler** | canvas 点击 → 坐标转换 → 格子状态判定 → 触发开地块 | `onClick(x, y)` |
| **GameLoop** | 10Hz 主循环，按顺序调用各系统 tick | `start()`, `stop()`, `tick()` |

### 技术选型

- **渲染：** Canvas 2D（零依赖，单文件友好）
- **游戏循环：** `requestAnimationFrame` + 固定时间步长（100ms/次，10Hz）
- **状态管理：** 单一 `GameState` 对象，不可变更新
- **六边形布局：** 平顶六边形（flat-top），axial coordinate (q, r)
- **随机数：** 单一 seeded RNG 实例贯穿所有随机源（混合池、档位、二次概率、开局），确保 replay 可复现
- **交付：** MuleRun 单文件 HTML，所有资源内联

### 格子状态机

```
[未激活] ──(首次相邻己方领地)──► [可开: 图标 + 价格]
                                      │
                                      │ (玩家点击 + 支付)
                                      ▼
                                [已开: 建筑结果]
                                      │
                          ┌───────────┼───────────┬──────────────┐
                          ▼           ▼           ▼              ▼
                      [兵营建筑]  [兵种卡]    [箭塔]         [金矿]
                       (吐兵)     (吐兵)      (防御)         (产金)
                          │           │           │              │
                          └───────────┴─────┬─────┴──────────────┘
                                            │ (HP 归零)
                                            ▼
                                        [废墟]
                                     (不可重建/不可通行)
```

另：兵营槽在 25 金档位有 50% 概率进入"已开但空"的终态（扣费但无建筑）。

### 关键交互流程

**开地块**：
1. 玩家点击 canvas → `ClickHandler` 转换为 axial 坐标 → `HexGrid.findHexAt` 命中判定
2. 查格子状态：若为"可开"，读其 `price` 和 `icon`
3. 校验 `ResourceSystem.trySpend(player, price)`：不足则 `showNotification("金币不足")`
4. 扣费成功后，按 `icon` + `price` 走 `TileSystem.open` 二次概率表 → 得到 `BuildingResult`
5. 若结果非空：更新格子为"已开 + 建筑"，注册到对应系统（兵营/箭塔/金矿）
6. 若结果空（25 金档 ? 特例）：更新格子为"已开但空"
7. 播放"开启"视觉反馈（粒子或闪光）

**推线连锁**：
1. `MoveUnits.tick` 让单位移动到新格子
2. 若新格子是敌方颜色：`TerritorySystem.claim(hexId, player)` 变色
3. 调用 `TileSystem.onTerritoryChanged()`：扫描所有未激活格子，若首次与己方领地相邻则激活（分配图标 + 价格）

**建筑摧毁连锁**：
1. `CombatSystem.tick` 检测到建筑 HP 归零
2. `RuinSystem.createRuin(hexId)` → 该格子变废墟
3. `TerritorySystem.onBuildingDestroyed(hexId)` → 该格子变为敌方颜色（敌方最后攻击方）
4. 当未来我方单位越过该废墟占领后方 → 废墟连锁变回我方颜色（但仍是废墟，不可重建）

**HQ 摧毁直胜**：
1. `CombatSystem` 检测到 HQ HP 归零 → 调用 `VictorySystem.onHQDamaged(player, lethal)`
2. `VictorySystem` 触发 0.5s 波浪扩散特效（`WaveEffect.spawn`），逐格将敌方格子变为我方颜色
3. 特效结束后显示胜利/失败弹窗 + "重新开始"按钮

### AI 决策算法

```
every 1.5s:
  actions = []
  for each openable tile in AI territory:
    actions.append(OpenTile(tile, price))
  for each AI building that can upgrade:
    actions.append(Upgrade(building, cost))
  actions.append(Wait)

  for action in actions:
    action.score = score(action, gameState)

  if difficulty == HARD:
    pick = argmax(actions, key=score)
  elif difficulty == NORMAL:
    pick = random.choice(top_k(actions, k=3, key=score))
  execute(pick)

score(action, state):
  return (
    0.3 * economy_gain(action) +        # 困难: 0.2, 普通: 0.3
    0.4 * roster_gap(action) +          # 困难: 0.4, 普通: 0.2
    0.3 * frontline_pressure(action) +  # 困难: 0.3, 普通: 0.2
    0.1 * expansion_opportunity(action) + # 困难: 0.1, 普通: 0.2
    noise * 0.1                         # 困难: 0.0, 普通: 0.1
  )
```

每 5 秒评估一次升级优先级（困难根据战况动态选，普通只升金矿）。

## Testing Decisions

### 测试标准

**好测试**的特征：
- 只测外部行为（输入 → 输出 / 状态变化），不测实现细节
- 用 seeded RNG 确保可复现
- 隔离模块边界：通过注入 mock GameState 测试单模块
- 覆盖边界条件：0 金币、HQ 0 HP、无合法动作、地图边界

**坏测试**的特征：
- 断言内部变量名或中间状态
- 依赖 wall-clock 时间（10Hz 下应使用 tickCount）
- 用 `Math.random` 而非 seeded RNG

### 测试覆盖范围

以下模块必须有独立的 `console.assert` IIFE 测试（沿用 v1 模式）：

- **HexGrid**：axialToPixel/pixelToAxial 互逆、getNeighbors 边界、isLake 判定、findHexAt 命中精度
- **RNG**：seeded 序列可复现、weightedPick 按权重分布（统计检验）
- **TileSystem**：状态机单向转换、混合池分布符合权重、二次概率表正确、激活判定正确
- **ResourceSystem**：addGold/trySpend 边界（负数、0、超出）、HQ+金矿 income tick
- **BarracksSystem**：兵营按 CD 吐兵、CD 计时、兵种品阶与建筑匹配
- **MoveUnits**：BFS 寻路到最近目标、己方穿越、飞行越格、品阶寻敌差异化
- **CombatSystem**：品阶压制伤害公式、远程/近战切换、AOE 溅射、建筑受击
- **TerritorySystem**：推线连锁、建筑摧毁连锁、占领计数
- **VictorySystem**：HQ 血量归零 → 直胜、倒计时归零 → 比格数、两种条件同时触发
- **AIBehavior**：评分函数在不同 gameState 下返回合理动作、难度参数影响决策分布
- **OpeningLayout**：≥1 金矿概率 70%、6 格随机分布

可暂不写独立测试的模块：DeathEffect、WaveEffect、UI/Render、ClickHandler（这些靠手工浏览器验证 + Playwright E2E）。

### 先验

v1 已有 9 个 embedded `console.assert` IIFE 测试（位于 index.html 底部）。v2 重写时**沿用同一测试模式**（`(function testXxx() { ... })()` 包裹，用 `console.assert` 断言）。v1 测试随 v1 模块一起删除，v2 测试在对应模块重写时逐步补齐。

### 已知陷阱（测试相关）

- **10Hz 下的时间断言**：`console.assert(tickCount > 0)` 同步失败，需 `setTimeout(..., 200)`
- **Seeded RNG 贯穿两层随机**：混合池 + 二次概率必须用同一个 RNG 实例，否则 replay 不可复现
- **金矿 income 干扰 "金币不足" 测试**：测试时设置 `mineCount=0` 禁用收入
- **Playwright file:// 被阻止**：E2E 必须用 `python3 -m http.server`

## Out of Scope

- **战前卡组系统**：v2 用"固定混合池"取代"玩家自建卡组"，MVP 不做卡组编辑
- **卡牌收集与升级**：元进度系统，MVP 不做
- **多阵营**：MVP 固定双方对战
- **单位合成 / 升星**：超出 v2 机制
- **实时 PvP**：MVP 只做 PvE
- **音频 / BGM**：MVP 不做
- **移动端适配**：MVP 只面向桌面浏览器
- **简单难度 AI**：v2 设计有三档难度，MVP 只实现普通 + 困难，简单档后续追加
- **建筑升级 UI 的复杂交互**：MVP 采用"点击已开建筑弹出升级面板"的最简方案，不做快捷键或批量升级
- **存档 / 复盘**：MVP 不做

## Further Notes

- **v1 测试与 v1 机制同时删除**：不要试图保留 v1 测试作为"回归保护"，v1 和 v2 机制层面互斥，保留会引发混淆。
- **数值调优后置**：v2 文档中多处标"待调优"的数值（混合池权重、二次概率表、AI 评分权重）在切片 1 完成后通过手动试玩调优，不要在编码阶段过度打磨。
- **单文件膨胀预警**：v1 是 1369 行，v2 预计 2500-3000 行。已接近单文件架构的合理上限，如果 v2 开发过程中发现还要扩，应警惕是否在做不该做的事。
- **Deadline 无缓冲**：25 天 5 切片 = 5 天/切片，若某切片卡住需果断砍 scope（比如先不做升级系统、先不做粒子特效），不要加 deadline。
- **AI 评分函数参数需要手工调优**：初版评分权重是拍脑袋的（经济 0.3/阵容 0.2/前线 0.2/扩张 0.2/噪声 0.1），困难档的调权（经济 0.2/阵容 0.4/前线 0.3/扩张 0.1/噪声 0.0）也是拍脑袋的。切片 5 做完后必须手工试玩调参，不能留到 deadline 当天。
- **种子一致性**：所有随机源（混合池、价格档位、二次概率、开局 6 格、AI 噪声）必须共享同一个 seeded RNG 实例，否则"重玩同 seed 结果不同"会严重伤害调试和 QA。
