# PRD: 占城大师 MVP 原型

## Problem Statement

作为独立游戏开发者，我需要在 3.5 周内（截止 2026-06-22）制作一个可玩的「占城大师」玩法原型。原版游戏在移动端/微信小游戏平台爆火，其核心玩法（六边形翻牌 + 自动出兵 + 资源管理 + 领地争夺）具有极强的可玩性和传播性，但目前市场上缺少类似的单文件 Web 版本可供快速体验和二次开发。

## Solution

开发一个零依赖的单文件 HTML 游戏（MuleRun 格式），包含完整的六边形地图翻牌、资源产出、兵营建造、自动出兵、3 兵种克制、AI 对手和胜负判定系统。玩家打开浏览器即可游玩，无需安装任何依赖。

## User Stories

1. 作为玩家，我想看到一张 5×7 的六边形地图，两端各有一个大本营，中间是迷雾地块，所以我能理解战场的整体布局。
2. 作为玩家，我想点击迷雾地块进行翻牌，翻牌后显示地块内容（金矿/空地/野怪/特殊），所以我能探索未知并获得资源或单位。
3. 作为玩家，我想在翻出金矿时获得一笔启动金币，并且金矿每秒持续产出金币，所以我能积累资源进行扩张。
4. 作为玩家，我想在已占领的空地上建造兵营（消耗金币），所以我能产生作战单位。
5. 作为玩家，我希望兵营自动产出随机兵种（步兵/骑兵/弓箭手），单位自动向敌方方向推进，所以我只需要关注地图决策而非微操。
6. 作为玩家，我想看到兵种之间存在克制关系（步>骑>射>步），克制方造成更高伤害，所以我能感受到策略深度。
7. 作为玩家，我想看到我的单位占领敌方或中立格子，格子颜色随占领方变化，所以我有"占城"的成就感。
8. 作为玩家，我想看到双方的金币数量实时显示在界面上，所以我能判断当前经济状况。
9. 作为玩家，我想看到单位交战时的伤害和血量变化，所以我能了解战斗进展。
10. 作为玩家，我想在摧毁敌方大本营时获得胜利提示，在己方大本营被摧毁时获得失败提示，所以我能明确游戏结果。
11. 作为玩家，我希望 AI 对手能独立翻牌、建兵营、出兵，形成有效对抗，所以我有真实的对手体验。
12. 作为玩家，我希望游戏能在约 3 分钟内结束，所以我有快节奏的体验。

## Implementation Decisions

### 模块划分

| 模块 | 职责 | 接口 |
|------|------|------|
| **HexGrid** | 六边形网格渲染、坐标计算、点击拾取 | `render()`, `getHexAt(x,y)`, `updateHex(hexId, state)` |
| **TileReveal** | 翻牌逻辑、地块内容生成 | `reveal(hexId) → TileData`, `generateMap(seed)` |
| **ResourceSystem** | 金币收入/支出管理 | `addGold(player, amount)`, `spendGold(player, amount)`, `tick()` |
| **BarracksSystem** | 兵营建造、单位产出调度 | `buildBarracks(hexId, player)`, `tick()` → spawn units |
| **CombatSystem** | 单位移动、交战、克制、死亡 | `tick() → combat results`, `resolveCombat(unitA, unitB)` |
| **TileOwnership** | 格子占领判定、阵营颜色 | `claim(hexId, player)`, `getOwner(hexId)` |
| **AIBehavior** | AI 翻牌/建兵营决策 | `tick(gameState) → actions[]` |
| **GameLoop** | 10Hz 主循环、胜负判定 | `start()`, `stop()`, `checkVictory()` |

### 技术选型
- **渲染：** Canvas 2D（零依赖，单文件友好）
- **游戏循环：** `requestAnimationFrame` + 固定时间步长（100ms/次，10Hz）
- **状态管理：** 单一 `GameState` 对象，不可变更新
- **六边形布局：** 平顶六边形，pointy-top 排列，坐标使用 axial coordinate (q, r)

### 核心数值
| 参数 | 值 | 说明 |
|------|-----|------|
| 地图大小 | 5 列 × 7 行 | 约 35 格 |
| 翻牌消耗 | 10 金 | 每次翻牌 |
| 兵营造价 | 40 金 | 建一个兵营 |
| 金矿启动资金 | 50 金 | 翻出金矿时立即获得 |
| 金矿持续产出 | 5 金/秒 | 每个金矿 |
| 兵营产出间隔 | 3 秒 | 每 3 秒产出一个单位 |
| 翻牌分布 | 40%金矿 / 30%空地 / 20%野怪 / 10%特殊 | 预置权重 |
| 步兵 | HP 220, ATK 15 | 克制骑兵 |
| 骑兵 | HP 180, ATK 18 | 克制弓箭手 |
| 弓箭手 | HP 120, ATK 20 | 克制步兵 |
| 克制倍率 | ×1.5 / ×0.5 | 克制方伤害 / 被克方伤害 |
| 交战频率 | 1 次/秒 | 每个单位每秒攻击一次 |

### 地块生成
地图生成使用固定种子（可配置），保证可复现。生成步骤：
1. 初始化 5×7 网格，两端设大本营（已翻开）
2. 中间 33 格按权重随机分配类型
3. 金矿格标记产出值，空地格标记可建兵营，野怪格标记难度

### 出兵方向
兵营产出的单位沿最短路径向敌方大本营方向推进，占领相邻的中立/敌方格子。

### AI 行为树（固定逻辑）
1. 每 2 秒尝试翻一块最近的未翻开格子（如果金币 ≥ 10）
2. 每 5 秒检查：如果金币 ≥ 40 且己方前线兵力 < 3，建一个兵营
3. 兵营自动出兵，无需 AI 额外决策

## Testing Decisions

### 测试策略
单文件原型不引入测试框架，采用以下策略：
- **手动验证：** 浏览器直接打开，逐个验证用户故事
- **核心逻辑单元测试：** 使用 `console.assert` 在代码中内嵌断言，验证 `CombatSystem.resolveCombat`、`ResourceSystem` 收支平衡、`HexGrid` 坐标计算

### 需验证的模块
| 模块 | 验证方式 | 验收标准 |
|------|----------|----------|
| HexGrid 渲染 | 浏览器截图 | 5×7 六边形网格正确渲染，无重叠/间隙 |
| TileReveal | 点击翻牌 | 点击迷雾格正确翻开，显示正确类型 |
| ResourceSystem | 金币数值观察 | 翻金矿后金币增加，翻牌消耗正确，金矿持续产出 |
| BarracksSystem | 建造观察 | 空地建兵营后，定时产出单位 |
| CombatSystem | 战斗观察 | 克制关系正确，HP 归零单位消失 |
| AIBehavior | AI 行为观察 | AI 定期翻牌、建兵营、出兵 |
| Victory | 胜负判定 | 大本营被摧毁时弹出胜负提示 |

## Out of Scope

以下内容明确不在 MVP 范围内：
- **卡牌收集/升级系统** — 长期养成，不影响核心循环
- **多阵营选择** — 不死族、机械等差异化阵营
- **单位合并/升星** — 增加战力深度的系统
- **PvP 实时对战** — 需要后端支持，违背单文件原则
- **音效/BGM** — 原型阶段不需要
- **广告/变现** — 不涉及商业化
- **移动端适配** — 先做桌面浏览器版本
- **动画特效** — 仅基础视觉反馈，不追求表现力

## Further Notes

1. **与《AI 副官》的关系：** 本项目独立于《AI 副官》主线，但兵种克制链、单局时长等设计参数保持一致，便于后续可能的整合。
2. **MuleRun 交付：** 最终产物为单文件 `index.html`，所有 CSS/JS/资源内联，浏览器直接打开即可运行。
3. **扩展性考虑：** 虽然 MVP 采用固定行为 AI 和预置地图，但模块设计预留了接口，后续可替换为更复杂的 AI 策略和动态地图生成。
4. **性能约束：** 单局约 3 分钟，Canvas 渲染 10Hz，单位总数控制在 30 以内，确保低端设备流畅运行。
